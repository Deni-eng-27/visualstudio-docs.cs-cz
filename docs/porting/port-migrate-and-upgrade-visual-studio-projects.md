# <a name="project-migration-and-upgrade-reference-for-visual-studio"></a>Odkaz na migraci a upgrade projektu pro Visual Studio

::: moniker range="vs-2017"

Každá verze sady Visual Studio obecně podporuje většinu předchozích typů projektů, souborů a dalších datových zdrojů. Můžete s nimi pracovat [jako vždy](../ide/solutions-and-projects-in-visual-studio.md)a za předpokladu, že nejste závislí na novějších funkcích, se Visual Studio pokusí zachovat zpětnou kompatibilitu s předchozími verzemi, jako je Visual Studio 2015, Visual Studio 2013 a Visual Studio 2012. (Viz [poznámky k verzi,](/visualstudio/releasenotes/vs2017-relnotes/) pro které jsou specifické funkce pro které verze.)

Podpora pro některé typy projektů se také v průběhu času mění. Novější verze sady Visual Studio již nemusí podporovat určité projekty vůbec nebo vyžaduje aktualizaci projektu tak, aby již není zpětně kompatibilní. Aktuální stav problémů s migrací najdete na [webu komunity pro vývojáře visual studio](https://developercommunity.visualstudio.com).

Tento tento článek obsahuje podrobnosti pouze pro typy projektů, které může aplikace Visual Studio 2017 migrovat. Článek vyloučí typy projektů, které již nejsou podporovány v sadě Visual Studio 2017 a proto nelze migrovat. Článek také vylučuje podporované typy projektů, které nemají žádné problémy s migrací; tento seznam se nachází na [platformě cílení a kompatibility](/visualstudio/productinfo/vs2017-compatibility-vs).

> [!IMPORTANT]
> Některé typy projektů vyžadují instalaci příslušných úloh prostřednictvím instalačního programu sady Visual Studio. Pokud nemáte nainstalované zatížení, Visual Studio hlásí neznámý nebo nekompatibilní typ projektu. V takovém případě zkontrolujte možnosti instalace a akci opakujte. Podrobnosti o podpoře projektů ve Visual Studiu 2017 najdete v článku [Cílení na platformu a kompatibilita.](/visualstudio/productinfo/vs2017-compatibility-vs)

## <a name="how-visual-studio-decides-when-to-migrate-a-project"></a>Jak visual studio rozhoduje o migraci projektu

Každá nová verze sady Visual Studio obecně usiluje o zachování kompatibility s předchozími verzemi, takže stejný projekt lze otevřít, upravit a sestavit v různých verzích. V průběhu času však dochází k nevyhnutelným změnám, takže některé typy projektů již nemusí být podporovány. (Viz [Cílení na platformu a kompatibilita,](/visualstudio/productinfo/vs2017-compatibility-vs) pro které jsou v Sadě Visual Studio 2017 podporované typy projektů.) V těchto případech novější verze sady Visual Studio nenačte projekt a nenabízí cestu migrace; je třeba udržovat tento projekt v předchozí verzi sady Visual Studio, která jej podporuje.

V ostatních případech novější verze sady Visual Studio můžete otevřít projekt, ale musí aktualizovat nebo migrovat projekt takovým způsobem, který může způsobit, že je nekompatibilní s předchozími verzemi. Visual Studio používá řadu kritérií k určení, zda je tato migrace nezbytná:

- Kompatibilita s cílovými verzemi platforem, zpět na Visual Studio 2013 RTM.

- Kompatibilita datových zdrojů v době návrhu s předchozími verzemi sady Visual Studio. (Konkrétně různé kanály Visual Studia 2017; Visual Studio 2015 RTM & aktualizace 3; Visual Studio 2013 RTM & aktualizace 5; Visual Studio 2012 Aktualizace 4; Visual Studio 2010 SP 1.) Visual Studio 2017 si klade za cíl selhat řádně s zastaralé prostředky návrhu bez jejich poškození, tak, aby předchozí verze můžete stále otevřít projekt.

- Určuje, zda by nové prostředky doby návrhu přerušily kompatibilitu s předchozími verzemi až do visual studia 2013 RTM & aktualizace 5.

Vlastník inženýrského typu daného typu projektu se těmito kritérii zabývá a provede volání, pokud jde o podporu, kompatibilitu a migraci. Visual Studio se opět pokusí zachovat transparentní kompatibilitu mezi verzemi sady Visual Studio, pokud je to možné, což znamená, že je možné vytvářet a upravovat projekty v jedné verzi sady Visual Studio a funguje pouze v jiných verzích.

Pokud taková kompatibilita není možná, ale stejně jako u některých typů projektů popsaných v tomto článku, pak Visual Studio otevře průvodce upgradem provést nezbytné jednosměrné změny.

Takové jednosměrné změny mohou `ToolsVersion` zahrnovat změnu vlastnosti v souboru projektu, což přesně označuje, která verze MSBuild může změnit zdrojový kód projektu na spustitelné a nasaditelné artefakty, které nakonec chcete. To znamená, co vykreslí projekt nekompatibilní s předchozími verzemi sady Visual Studio není verze `ToolsVersion`sady Visual *Studio,* ale verze *MSBuild,* jak je určeno . Tak dlouho, dokud vaše verze sady Visual Studio obsahuje `ToolsVersion` řetězec nástrojů MSBuild, který odpovídá v projektu, pak Visual Studio můžete vyvolat tento řetězec nástrojů k sestavení projektu.

Chcete-li zachovat maximální kompatibilitu s projekty vytvořenými ve starších verzích, `ToolsVersion` Visual Studio 2017 obsahuje potřebné msbuild toolchains pro podporu 15, 14, 12 a 4. Projekty, které `ToolsVersion` používají některou z těchto hodnot by mělo mít za následek úspěšné sestavení. (S výhradou, opět, zda Visual Studio 2017 podporuje typ projektu vůbec, jak je popsáno na [platformě cílení a kompatibility](/visualstudio/productinfo/vs2017-compatibility-vs).)

V tomto kontextu přirozeně vyvstává otázka, zda byste se měli pokusit `ToolsVersion` ručně aktualizovat nebo migrovat projekt na novější hodnotu. Provedení takové změny je zbytečné a pravděpodobně by generovat mnoho chyb a upozornění, které budete muset opravit získat projekt znovu sestavit. Kromě toho pokud Visual Studio klesne `ToolsVersion` podporu pro konkrétní v budoucnu, pak otevření `ToolsVersion` projektu spustí proces migrace projektu konkrétně proto, že hodnota musí být změněna. V takovém případě subsystém pro tento konkrétní typ projektu přesně ví, co je třeba změnit, a můžete tyto změny provést automaticky, jak je popsáno výše v tomto článku.

## <a name="migrate-a-project"></a>Migrace projektu

Zatímco se snažíme zachovat kompatibilitu s předchozími verzemi, mohou existovat změny, které nejsou kompatibilní s předchozími verzemi. (Viz [Cílení na platformu a kompatibilita,](/visualstudio/releases/2019/compatibility) pro které typy projektů jsou podporovány v Sadě Visual Studio 2019.) V takovém případě novější verze sady Visual Studio nebude načíst projekt nebo nabídnout cestu migrace. Pravděpodobně bude muset udržovat tento projekt v předchozí verzi sady Visual Studio.

V některých případě novější verze sady Visual Studio můžete otevřít projekt, ale musí aktualizovat nebo migrovat projekt způsobem, který může způsobit, že je nekompatibilní s předchozími verzemi. Visual Studio používá řadu kritérií k určení, zda je tato migrace nezbytná:

- Kompatibilita s cílovými verzemi platforem, zpět na Visual Studio 2013 RTM.

- Kompatibilita datových zdrojů v době návrhu s předchozími verzemi sady Visual Studio. (Konkrétně různé kanály Visual Studia 2019, Visual Studio 2017; Visual Studio 2015 RTM & aktualizace 3; Visual Studio 2013 RTM & aktualizace 5; Visual Studio 2012 Aktualizace 4; Visual Studio 2010 SP 1.) Visual Studio 2019 si klade za cíl selhat elegantně s zastaralé prostředky návrhu bez jejich poškození, tak, aby předchozí verze můžete stále otevřít projekt.

- Určuje, zda by nové prostředky doby návrhu přerušily kompatibilitu s předchozími verzemi až do visual studia 2013 RTM & aktualizace 5.

Technický tým, který vlastní typ projektu, se na tato kritéria podívá a provede volání, pokud jde o podporu, kompatibilitu a migraci. Znovu se snažíme zachovat kompatibilitu mezi verzemi sady Visual Studio tak, aby při vytváření a úpravách projektů v jedné verzi sady Visual Studio, funguje pouze v jiných verzích.

Někdy kompatibilita není možná. Potom Visual Studio otevře průvodce upgradem provést potřebné jednosměrné změny. Tyto jednosměrné změny může `ToolsVersion` zahrnovat změnu vlastnosti v souboru projektu, který přesně označuje, která verze MSBuild můžete proměnit zdrojový kód projektu do spustitelné a nasaditelné artefakty, které chcete. 

Co vykreslí projekt nekompatibilní s předchozími verzemi sady Visual Studio není verze sady `ToolsVersion`Visual *Studio,* ale verze *MSBuild,* jak je určeno . Pokud vaše verze sady Visual Studio obsahuje řetězec `ToolsVersion` nástrojů MSBuild, který odpovídá v projektu, pak Visual Studio můžete vyvolat tento řetězec nástrojů k sestavení projektu.

Chcete-li zachovat kompatibilitu s projekty, které jste vytvořili v předchozích verzích, Visual Studio 2019 obsahuje potřebné MSBuild toolchains pro podporu `ToolsVersion` 15, 14, 12 a 4. Projekty, které `ToolsVersion` používají některou z těchto hodnot by mělo mít za následek úspěšné sestavení. (S výhradou, znovu, zda Visual Studio 2019 podporuje typ projektu, jak je popsáno na [platformě cílení a kompatibility](/visualstudio/releases/2019/compatibility).)

Může být v pokušení ručně aktualizovat nebo migrovat `ToolsVersion` projekt na novější hodnotu. Je nutné provést takovou změnu a pravděpodobně by generovat mnoho chyb a upozornění, které je nutné opravit získat projekt znovu sestavit. Také pokud Visual Studio nepodporuje `ToolsVersion` konkrétní v budoucnu, pak projekt spustí proces migrace projektu `ToolsVersion` při jeho otevření, protože jeho hodnota musí být změněna.
